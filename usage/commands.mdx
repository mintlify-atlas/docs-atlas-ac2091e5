Laser supports two types of special commands in interactive mode: **slash commands** for controlling the session, and **shell commands** for running local commands directly.

## Slash Commands

Slash commands start with `/` and control Laser's session state. They are only available in interactive mode.

### `/quit` - Exit Laser

Exits the interactive session:

```
laser > /quit
Goodbye sad computer making shutdown noises...
```

Alternatively, press `Ctrl+C` to exit.

---

### `/reset` - Clear History

Clears the entire conversation history and starts fresh:

```
laser > /reset
┌─────────────────────────────────────────┐
│ /reset                                  │
│ Conversation reset. History cleared.    │
└─────────────────────────────────────────┘
```

**Use cases:**

- Starting a completely different task
- Reducing context size when the conversation gets very long
- Clearing sensitive information from history
- Starting over after going down the wrong path

<Note>
  After `/reset`, Laser has no memory of previous messages. You'll need to provide context again.
</Note>

---

### `/usage` - Show Token Usage

Displays cumulative token usage statistics for the current session:

```
laser > /usage
┌──────────────────────────────────────────────────┐
│ /usage                                           │
│ Token usage (cumulative)                         │
│                                                  │
│ Input :    45230  (other=12450, cache_read=28900│
│            cache_create=3880, cached=72.5%)      │
│ Output:     8420                                 │
│                                                  │
│ Total :    53650                                 │
└──────────────────────────────────────────────────┘
```

**Statistics explained:**

- **Input tokens**: All tokens sent to the model
  - `other`: New input (user messages, system prompt, tool results)
  - `cache_read`: Tokens read from prompt cache (cost savings)
  - `cache_create`: Tokens written to cache
  - `cached`: Percentage of input served from cache
- **Output tokens**: Tokens generated by the model
- **Total**: Sum of all input and output tokens

<Tip>
  Use `/usage` to monitor costs, especially with expensive models. Cache hits significantly reduce costs for repeated context.
</Tip>

---

### `/plan` - Toggle Planning Mode

Toggles planning mode on or off for subsequent tasks:

```
laser > /plan
┌────────────────────────────────────┐
│ /plan                              │
│ Planning is now enabled.           │
└────────────────────────────────────┘

laser > Add new feature
$: mkdir -p plans
$: cat > plans/add_new_feature_plan.md
...

laser > /plan
┌────────────────────────────────────┐
│ /plan                              │
│ Planning is now disabled.          │
└────────────────────────────────────┘
```

When enabled, Laser creates a checklist plan file before executing each task.

See [Planning Mode](/usage/planning-mode) for details.

## Shell Commands

Shell commands start with `!` and execute directly in your working directory, **bypassing the agent**.

### Syntax

```
!<command>
```

The command runs in a subprocess and prints stdout/stderr directly.

### Examples

<Tabs>
  <Tab title="Run Tests">
    ```
    laser > !pytest -q
    .....................                                    [100%]
    21 passed in 2.45s
    ```
  </Tab>
  
  <Tab title="Check Git Status">
    ```
    laser > !git status -sb
    ## main...origin/main
     M src/api/auth.py
    ?? plans/add_validation_plan.md
    ```
  </Tab>
  
  <Tab title="View File">
    ```
    laser > !cat src/config.py
    import os
    
    API_KEY = os.getenv("API_KEY")
    DEBUG = os.getenv("DEBUG", "false") == "true"
    ```
  </Tab>
  
  <Tab title="Run Linter">
    ```
    laser > !ruff check src/
    All checks passed!
    ```
  </Tab>
</Tabs>

### How Shell Commands Work

From the implementation in `main.py:309-332`:

1. **Detection**: If input starts with `!`, it's treated as a shell command
2. **Extraction**: The `!` prefix is stripped
3. **Execution**: Command runs via `subprocess.run()`
   - `shell=True` — supports pipes, redirection, etc.
   - `capture_output=True` — captures stdout and stderr
   - `check=False` — doesn't raise on non-zero exit codes
4. **Output**: Both stdout and stderr are printed
5. **No agent involvement**: The command bypasses Laser entirely

<Note>
  Shell commands are executed **locally** in your working directory. They are not sent to the LLM and don't use the `bash` tool.
</Note>

### Shell vs. Agent Bash Tool

<Tabs>
  <Tab title="Shell Command (!cmd)">
    ```
    laser > !rg "TODO" src/
    ```
    
    **Characteristics:**
    - Executes immediately
    - Bypasses the agent
    - Not part of conversation history
    - Full shell features (pipes, redirection, etc.)
    - No retry or error handling
  </Tab>
  
  <Tab title="Agent Tool (via message)">
    ```
    laser > Search for TODO comments in the src directory
    $: rg -n "TODO" src/
    ```
    
    **Characteristics:**
    - Agent decides when/how to run it
    - Becomes part of conversation context
    - Agent can interpret results
    - Can be retried or adjusted
    - Subject to agent's safety rules
  </Tab>
</Tabs>

### Examples

#### Combining Shell Commands with Agent Tasks

```
laser > Add type hints to the auth module
$: rg -n "def " src/auth.py
$: cat src/auth.py
...
I've added type hints to all functions in src/auth.py.

laser > !mypy src/auth.py
src/auth.py:42: error: Incompatible return value type
Success: type checked 1 file, 1 error found

laser > Fix the mypy error on line 42
$: cat src/auth.py
...
Fixed! The function now returns the correct type.

laser > !mypy src/auth.py
Success: type checked 1 file, 0 errors found
```

#### Quick Iteration Loop

```
laser > Refactor the database connection logic
...

laser > !pytest tests/test_db.py
FAILED tests/test_db.py::test_connection - TypeError: ...

laser > The test failed with a TypeError, can you fix it?
...

laser > !pytest tests/test_db.py
PASSED tests/test_db.py::test_connection
```

## Implementation Details

### Command Parsing

From `main.py:302-378`, Laser checks user input in this order:

1. **Shell commands** (`!...`) — line 309
2. **Slash commands** (`/reset`, `/usage`, `/plan`, `/quit`) — line 334-378
3. **Regular messages** — everything else

### Usage Tracking

The `UsageTracker` class (lines 110-146) accumulates tokens across all interactions:

```python
class UsageTracker:
    def __init__(self) -> None:
        self.input_other = 0
        self.input_cache_read = 0
        self.input_cache_creation = 0
        self.output = 0
```

It provides:
- Per-category tracking (regular input, cache reads/writes, output)
- Automatic cache percentage calculation
- Formatted display via `/usage` command

<Tip>
  Usage tracking persists for the entire session. Use `/reset` to clear history, but note that `/reset` does **not** reset the usage counter.
</Tip>

## Best Practices

### When to Use Each Command Type

**Slash commands:**
- Session management and control
- Checking statistics
- Changing modes (planning on/off)

**Shell commands:**
- Quick verification
- Running tests manually
- Checking git status
- One-off operations

**Regular messages:**
- All coding tasks
- Complex operations
- Anything requiring agent reasoning

### Common Workflows

<Steps>
  <Step title="Start fresh">
    ```
    laser > /reset
    ```
    Begin a new task with clean context.
  </Step>
  
  <Step title="Enable planning for complex work">
    ```
    laser > /plan
    laser > Implement new authentication system
    ```
  </Step>
  
  <Step title="Verify changes locally">
    ```
    laser > !git diff
    laser > !pytest tests/
    ```
  </Step>
  
  <Step title="Check resource usage">
    ```
    laser > /usage
    ```
    Monitor costs and optimize if needed.
  </Step>
</Steps>

<Note>
  Shell commands (`!`) only work in **interactive mode**. In non-interactive mode (`--prompt`), use regular messages and let the agent handle commands via its bash tool.
</Note>
